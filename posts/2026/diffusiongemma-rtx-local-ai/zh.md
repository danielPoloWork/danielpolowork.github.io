十年来，运行大语言模型的故事，归根结底是一个等待内存的故事。NVIDIA 在 RTX AI Garage 上那篇关于加速 Google DeepMind 的 DiffusionGemma 的文章之所以有意思，并不是因为它又宣布了一个快模型，而是因为它悄悄地把这个前提颠倒了过来。当你改变模型吐出文本的*方式*——从一次一个 token 变成一次一整块——你就改变了机器的哪个部分成为瓶颈。而一旦你挪动了瓶颈，原本闲置的硬件就成了胜出的硬件。我想把这个反转讲清楚，因为「快 4 倍」这个标题背后藏着的，才是真正的新闻。

## 深入解析

让我从一个新手该起步的地方讲起，因为一旦你不再把文本想成一条流，这个机制其实非常简单。

你用过的几乎每一个模型——那些让你感觉它在朝你*打字*的——都是**自回归的（autoregressive）**。它生成一个 token，看一眼自己迄今为止产出的全部内容，再生成下一个，如此往复。每个词字面意义上都依赖于它前面的那个词。这种依赖正是那种打字机感的来源，也是我稍后要回到的那个深层低效的来源。

据该来源，DiffusionGemma 走的是图像模型走过的路。一个扩散式图像模型从噪声出发，一次性地、一步一步地精修整幅画面，直到画面变得清晰。DiffusionGemma 对*文本*做同样的事：它从一块垃圾 token 出发，每一步**并行地对至多 256 个 token 去噪（denoise）**，把整块精修成连贯的语言，而不是吐出一个词、再等着算下一个。这个模型「以块为单位思考，而非逐个顺序地思考」。这一句话就是全部的想法——向新手解释它，就是*一次性填好一整段、然后再把它理顺的自动补全，而不是一个字母一个字母地敲出来*。

底下的架构值得钉牢，因为它解释了为什么这件事到现在才变得可行。该来源称 DiffusionGemma 构建于 **Gemma 4 之上——一个 260 亿参数的混合专家（MoE）模型，每一步只激活 38 亿参数**，在 Gemma 4 的 transformer 上加装了一个扩散头。混合专家的设计在这里很关键：你为一个 260 亿模型的质量付费，但在任意一步只让其中 38 亿参与计算。把这种俭省的每步计算与并行去噪配在一起，你就得到一个每步成本足够小、以至于一次处理 256 个 token 在单块 GPU 上也可行的模型。各部件咬合在一起：MoE 让每步的运算便宜，扩散让每步变宽，二者相抵，结果是「异常快速的文本生成」。

现在说反转，这是我觉得真正有启发的部分。该来源给出的框架很精确——一次一个 token 地生成是**一个内存受限（memory-bound）的问题**。传统 LLM「把大部分时间花在等待内存带宽上，而不是在做运算」。想想为什么——为了产出单单一个下一 token，GPU 必须把相关权重从内存里流出来，做相对而言极少量的算术，然后为下一个 token 把这一切再来一遍。昂贵的硅片——为做海量稠密矩阵运算而生的 Tensor Core——大多闲着，被内存总线饿着。这就是「被留在桌上的大量算力」，也是单用户本地推理一向感觉比它本该有的速度更慢的结构性原因。

扩散，用该来源的话说，「把方程式翻了过来」。把一整块 256 个 token 并行地推过 transformer，是一个**计算受限（compute-bound）的工作负载**——有一大堆稠密矩阵运算要一次性做完。而这恰恰是 GPU 的用途所在。该来源对机制的说法很直接——**NVIDIA Tensor Core 加速稠密的并行运算**，CUDA 软件栈让模型从第一天起就高效运行，无需定制调优。所以 Tensor Core 在这里的角色不是附带的——它就是要点本身。自回归递给 GPU 一个它不擅长的问题（等内存）；扩散递给它一个它本就为之设计的问题（稠密并行的乘加）。这个模型的形状，也许是无意间，被画成了贴合芯片的样子。

该来源报告的数字，正是从这种贴合中得出的——**单块 H100 上每秒 1,000 个 token**，DGX Spark 上每秒 150 个，以及 **DGX Station 上至多每秒 2,000 个**——「比在同一单用户体制下运行的等效自回归模型大约快 4 倍」。请记住那个限定语——*单用户*——因为整个论证都栖身于此，辩论也从这里开始。

## 辩论

第一个该摆上桌的问题，是怀疑者会立刻抛出的那个——**并行地生成一整块，能否保住顺序生成那种文本的质量？**

诚实的回答是：这个来源没有告诉我们。它是一份加速与部署的公告——报告吞吐、硬件支持和许可，而不报告对自回归基线的基准准确率。所以我会去推理，而不是断言。自回归严格的从左到右的依赖不只是一种性能税；它也是一种连贯性机制——每个 token 都以一个已完全确定的过去为条件。扩散把这一点拿去换并行性：它精修一整块，其中的 token 是带着双向上下文一起被决定的。原则上那种双向视野是一种*优势*——模型可以让一句话的结尾去影响它的开头。但实际上，扩散式语言模型在历史上一向用一些保真度换速度，举证责任落在该来源没有纳入的质量基准那一边。一位资深读者应当把「快 4 倍」当作可信，把「一样好」当作**此处未经证明**——貌似合理、在架构上有趣，但在这份材料里没有被实证。

第二场辩论更尖锐，也是该来源半埋起来的那个——**块生成究竟会改变本地 AI 的性能，还是只改变*单用户*本地 AI 的性能？** 那个 4 倍的数字被明确地限定在「单用户体制」。这个限定不是脚注——它就是论点本身。在云端服务的场景里，一个自回归模型把数百个用户的请求批在一起处理，而*正是*那种批处理把 Tensor Core 喂饱了——内存受限的代价被摊薄到整批之上。云端早就用并发解决了「GPU 闲着」的问题。扩散的并行性，则为云端解决不了的那种情形把它解决了——**坐在自己机器前的那个单一用户**，没有可供藏身的批。所以正确的读法不是「扩散比自回归快」。而是「扩散把数据中心级的 GPU 利用率，带给了数据中心从未为之优化的那唯一一种工作负载——你，独自一人，在自己的硬件上」。在我看来，这是整个故事里最重要的一句话。

这就引出了第三场、也是后果最重大的一场辩论——**这对本地助手和 AI 智能体意味着什么？** 该来源反复回到「智能体循环」和「能规划并行动的设备端助手」，而这种关联并非营销话术。智能体不是聊天机器人——它是一个循环。它生成一个计划，调用一个工具，读取结果，生成下一步，再绕一圈——为了一个用户可见的动作，往往要绕几十次。那个循环的每一圈都要付出生成的全部延迟。在本地硬件上的自回归之下，这种延迟层层累积，直到智能体感觉迟钝、用户放弃。把每次生成从缓慢的顺序爬行折叠成快速的并行块，循环就收紧到一个本地智能体能够「跟上开发者思考与迭代的节奏」的程度。延迟对智能体而言不是一个让人舒服的功能——它是在交互死掉之前、你能负担得起多少步推理的那个闸门。把它砍掉 4 倍，你改变的就不只是它是否惬意，而是本地*可能做什么*。

还有一处，我会论证它在战略上嗓门最大，尽管该来源几乎是顺带提了一句——**Apache 2.0、开放权重、不需云端、没有按 token 计费。** 把一个开放许可、能在带 128GB 统一内存的 GB10 Grace Blackwell DGX Spark 上——或在 RTX PRO 6000 工作站上，乃至 llama.cpp 支持落地后在 GeForce RTX 上——完全运行的模型组合起来，把每一步智能体动作都送进按量计费的云端 API 的那套经济账，就开始动摇。*在本地足够快*与*按 token 免费*的组合，才是真正可能推动这个领域的东西，而不是那个原始的 token 数。

## 评析

作为一篇沟通文本，该来源在硬件方面清晰、具体得令人称道——但作为读者，我必须点名它略去了什么，因为这些略去左右着那些主张能承载多大的分量。

第一，也是最重要的——**没有任何质量评估。** 没有一项针对它在速度上所比较的自回归基线的准确率、推理或连贯性基准。对于一个全部前提就是「换一种方式生成」的模型来说，那个没被回答的问题是：换一种方式生成，在正确性上是否要付出代价。该来源要我们在没有展示质量被保持恒定的情况下接受一场速度上的胜利，而那恰是技术读者最需要的那个数字。我会催促作者把一张正面对决的准确率表放在吞吐表旁边发布。

第二，吞吐数字虽然具体，却**条件不足**。H100 上每秒 1,000 个 token——在多长的序列、多大的批、什么精度下？Hugging Face 的链接点名了一个 **NVFP4** 量化版本；如果那个 4 倍的数字是骑在 4 位量化之上的，那就是一个有分量的附注，因为量化本身就是一根质量的杠杆。没有测量条件的速度主张只是方向性的，而非可复现的。

第三——这是一条框架上的批评，而非事实上的——这篇文章是由一家硬件厂商发布的，而且看得出来。叙事的弧线弯向「这种工作负载爱我们的芯片」，那是*真的*、也论证得好，但它意味着比较永远是 NVIDIA 上的扩散对 NVIDIA 上的自回归。文中没有讨论扩散式语言模型仍然吃力的地方（可变长度输出、停止判据、对超出块大小的超长生成的处理），一个利益没那么深的来源会去触碰那些边缘。那条由无关产品新闻组成的「#ICYMI」尾巴，进一步印证了这在结构上是——一篇内里裹着异乎寻常优质工程内容的——营销。

为了对这份材料公平：它的核心技术主张——从内存受限到计算受限的反转，以及由此对 Tensor Core 的贴合——是**健全且解释得当的**，不是炒作。那个论证经得起同行审视。我的批评不是说这个故事是错的——而是说，恰恰在那个有人把生产系统押上去之前需要独立评估的维度（质量、测量的严谨）上，这个故事是*不完整的*。

## 摘要

DiffusionGemma 像图像模型生成图片那样生成文本：每一步并行地对一块至多 256 个 token 去噪，而非一次吐出一个。它在 Gemma 4 的 260 亿 MoE 模型（每步激活 38 亿参数）上加一个扩散头，以 Apache 2.0 开放发布，报告单用户生成至多快 4 倍——H100 上每秒 1,000 个 token，DGX Station 上至多 2,000 个。

那个深层的想法是一次反转：自回归是内存受限的，让 NVIDIA 的 Tensor Core 闲着；并行块扩散是计算受限的，正好贴合这些核心在稠密并行运算上的强项。那个「4 倍」明确是一场*单用户*的胜利——扩散把数据中心级的 GPU 利用率，带给了云端从未为之优化的本地、无批处理的工作负载。那才是可能重塑本地 AI 的东西：收紧的、低延迟的智能体循环，以及运行飞快、不受按 token 计费束缚的设备端助手。

该来源没有回答的悬而未决的问题是质量——没有准确率基准，吞吐数字缺测量条件，而一个 NVFP4 量化版本可能正扛着那个速度主张。把速度当作可信，把「一样好」当作未经证明。架构与硬件的贴合是那个真正的洞见；与自回归质量的对等，则是那场仍在等待被展示的实验。
