在 Gemma 4 12B 论证「本地机器配得上一套自己的模型架构」一周之后，Google 出货了这一论点更激进的版本。我想审视的资料——实验性的 26B Mixture-of-Experts 模型 **DiffusionGemma** 的发布说明——保留了每一个主流 LLM 的自回归主干，却抛弃了它们共有的那唯一一件事：从左到右、一次一个 token 的解码。它以扩散方式一次生成 256 个 token，宣称输出速度最高快 4 倍。作为一个以剖析本地推理为生的人，我把这个头条数字读作最无趣的部分。有趣的部分是：*它挪动的是哪一道瓶颈。*

## 深入解析

统辖整个发布的那一句话是这个：DiffusionGemma 之所以奏效，是「通过把解码瓶颈从内存带宽挪到计算」。请在此驻足，因为这就是全部论点，而大多数报道会跳过它直奔 4 倍。

当你为单个用户在本地运行一个自回归模型时，你的 GPU 并非计算受限——而是*带宽*受限。生成一个 token，意味着把模型权重从 HBM 流式取出，做相对而言极少量的算术，然后等待。在批大小为 1 时，H100 那巨大的 FLOPS 大多闲置，而内存总线成了约束。资料自身的比喻很贴切：打字机把大部分时间花在等待下一次敲键上。这正是单用户本地推理让人觉得在浪费你买来的昂贵加速器的原因——它确实在浪费。

扩散把这个体制反转过来。DiffusionGemma 不再逐词预测，而是一次性起草整整 256 个 token 的块：它从一张随机占位 token 的画布出发，做多轮精炼，锁定有把握的 token，并以它们为上下文去修正其余，直到该块收敛。把一大块并行工作一次性交给处理器，恰恰是让闲置 FLOPS 饱和的办法。所以这 4 倍不是免费午餐，也不是一个更小的模型——它是*对一项未被充分利用资源的套利*。你本就在为计算付费，是自回归一直不让你用上它。

那个框架立刻告诉你速度提升栖身于何处、又在何处消亡，而说明对二者都坦白得令人钦佩。优势在单个加速器上的中低批大小——本地、低并发的情形——最强。在高 QPS 的云端服务中，画面*反转*了：自回归模型通过把数千个请求一起批处理，本已使计算饱和，于是扩散的并行解码变得冗余，服务成本反而可能更高。因此 DiffusionGemma 并非抽象地「更快」。它在一个特定的工作点——你的，在你的桌上——更快，而在云端那个工作点上以成本计更慢。经济性随并发而反转。这是比「4 倍」更精细的主张，也是正确的那个。

两项设计选择让这在消费级硬件上变得可行。MoE 结构把存储的容量与每一步的成本解耦：总参数 26B，但每次前向只激活 3.8B。而量化——Google 与 NVIDIA 合作，做出了「近乎无损精度」的 4 比特浮点格式 NVFP4——把常驻占用缩小，装进像 RTX 5090 这样高端消费 GPU 的 18GB VRAM。不过要精确地看清各自买来什么，因为说明在此略有含糊——VRAM 占用受*总*参数制约（你必须持有整个量化后的 26B），而吞吐受*激活*参数制约（每轮只计算 3.8B）。MoE 是每 token 速度的杠杆，量化是装进内存的杠杆。两者合力，在 H100 上跑出每秒 1000+ token，在 RTX 5090 上跑出 700+。

但资料中最深的一点根本与速度无关，且被埋在一条微调脚注里。由于全部 256 个 token 以**双向注意力**一同生成，每个 token 都能注意到所有其他 token——包括「在它之后」的那些。自回归模型是因果掩码的：token N 被禁止看到 token N+1，因此必须在知道后面是什么之前就拍板。对于任何其正确性是*全局*约束、而非从左到右序列的任务，那道掩码都是一种结构性的劣势。Sudoku 演示（Unsloth 把 DiffusionGemma 微调来解数独）是干净的例证——自回归模型之所以吃力，恰恰因为每个格子都依赖于未来的格子。同样的逻辑延伸到代码填充、氨基酸序列与数学图。这不是一个速度特性，而是一项自回归从根本上缺失的能力。

## 辩论

那么让我毫不退缩地论辩这真正的张力——速度对品质——因为说明也没有退缩。

**支持扩散：** 延迟上的胜利是真实的，而且它瞄准的正是延迟即产品的工作流。行内编辑、代码填充、快速迭代、代码与 markdown 的近实时渲染——这些都是开发者*在等待光标*的交互循环。把生成从逐字打字机的速度削减到印刷机的速度，会改变这工具用起来的手感。而双向能力还添上了自回归在任何速度下都做不到的东西：同时满足全局约束。对于一个构建本地、单用户、交互式工具的开发者而言，这比一个更快的自回归模型是一件形状更好的器械。

**支持自回归：** 但说明用平实的语言陈明了代价——「整体输出品质低于标准 Gemma 4」，且为求最高品质，他们推荐部署标准 Gemma 4。这不是一句保留，这就是那笔权衡。你在花品质去买延迟。迭代式的自我修正（模型回读整个块以纠正自身错误）找回一些，但公司自己的推荐告诉你前沿落在哪里。对于正确性占主导的生产输出，打字机仍然胜出。

**开发者实际应当抵达的综合**是：「更好」是错误的框架，*工作点*才是正确的框架。DiffusionGemma 是一步帕累托之举，而非严格的改进——它在品质上让步，以换取本地延迟，以及一项自回归够不着的能力（双向推理）。决策规则干净地导出——交互式、本地、单用户、延迟受限，或全局约束的任务 → 扩散；高品质、生产、批处理，或云端服务的任务 → 自回归。正确的心智模型不是「新模型取代旧模型」，而是「Google 如今为两台不同的计算机出货两个解码器」，一套认真的本地技术栈大可两者都跑，按任务分流。

## 评析

作为面向开发者的沟通，说明在对自身限度坦诚时最为出色——而这比大多数发布帖都更频繁。

从内存带宽到计算的框架既出色又罕见；它给工程师的是真正的心智模型，而非一个营销数字，而那句明确承认「优势在云端反转」的话，是建立信任的那类坦率。打字机到印刷机的比喻立得住，却略微低估了机制——印刷机刷一次，而扩散做*多轮*精炼。因此那个净的 4 倍已是这些轮次之后的值，这一细节值得点明，因为它正是品质权衡的来源。

有两件事被低估了。其一，也是最重要的，双向注意力这项能力是整个发布里最新颖的东西，却被降格为一个关于 Sudoku 的微调示例。头条是一个速度倍率；而导语本应是「自回归模型看不见自己未来的 token，对于一整类逻辑与结构任务，那是一堵墙——这是一个没有那堵墙的模型」。那才是持久的贡献；4 倍是一个会变旧的数字。

其二，品质权衡虽被承认，却在最要紧处没有被*量化*。我们被告知品质更低，被展示一张「Intelligence vs Latency」的图与一张基准条形图，但开发者用以下判断所需的唯一一样东西——在哪些任务上、为多少速度、付出多少品质——却没有给出。我在 Gemma 4 12B 说明里指出的同一道隙缝在此重现——部署面被充满爱意地详述（MLX、vLLM、Transformers、NeMo、Hackable Diffusion、横跨消费与 DGX 硬件的 NVFP4 内核），而代价那一侧却以图表出货。一个整个卖点就是*权衡*的模型，欠它的受众一个汇率。

最后，一处虽小却真实的遗漏——固定的 256 token 块引出显而易见的问题：更长的生成中块如何串接、那 4 倍假定了多少轮精炼、延迟如何随输出长度扩展。说明把这些留给了开发者指南。对于一篇主题本身就是延迟的文字而言，延迟曲线本身却缺席了。

## 摘要

DiffusionGemma 是一个实验性的 26B-MoE（激活 3.8B）开放模型，它以文本扩散取代自回归的逐 token 解码：并行起草 256 个 token 并迭代精炼，带来最高 4 倍的生成提速（H100 上每秒 1000+ token，RTX 5090 上 700+），并经 NVFP4 量化装进 18GB 的消费级 VRAM。真正的洞见是机制性的——它把本地解码瓶颈从内存带宽挪到计算，把你闲置的加速器 FLOPS 变成速度——而那项优势是本地、低并发推理所特有的；在高 QPS 的云端服务中它反转，且成本更高。权衡是明确且为公司所承认的：输出品质低于标准 Gemma 4，而后者仍是生产品质的推荐。最被低估的部分不是速度，而是双向注意力，它让每个 token 都能注意到所有其他 token，解锁了自回归在结构上无法完成的全局约束任务——数独、代码填充、结构化序列。对开发者而言，正确的读法不是取代而是工作点——Google 如今为两台不同的机器出货两个解码器，而一套深思熟虑的本地技术栈，会把交互式、延迟受限、约束繁重的工作分流给扩散，并把自回归留给品质即产品之时。
