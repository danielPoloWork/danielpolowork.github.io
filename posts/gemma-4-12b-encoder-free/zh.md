在多模态时代的大部分时间里，我们一直以同样的方式构建这些系统——取一个语言模型，在它一侧拼接一个单独训练的视觉编码器，在另一侧拼接一个音频编码器，再学习一个在二者之间翻译的投影。我想审视的资料——为开发者撰写的 **Gemma 4 12B** 发布说明——静静地抛弃了这一范式。它移除了编码器。作为一个出货推理流水线的人，我读它，与其说是一个基准故事，不如说是一项对眼前这台机器上能跑什么有着直接后果的架构决定。

## 深入解析

从传统设计说起，因为这条新闻只有对照它才说得通。一个传统的多模态模型，实质上是披着风衣的三个训练产物：一个视觉编码器（通常以对比式视觉语言目标预训练的 ViT）、一个音频编码器（以转录训练的 Whisper 级网络），以及 LLM 主干——由把每个编码器的输出映射到主干嵌入空间的投影层缝合在一起。每个编码器都是一个强大而专门的先验。但它同时也是你必须加载的权重、必须支付的延迟，以及流水线里事物会崩坏的那道接缝。

Gemma 4 12B 把这道接缝折叠掉了。对于**视觉**，说明称编码器被替换为「一个由单次矩阵乘法、位置嵌入与归一化构成的轻量嵌入模块」，由 LLM 主干接管视觉处理。请精确地读：那个专门的感知网络消失了，缩减为与一个分块投影相差无几的东西，而表征的负担移到了主干*内部*。对于**音频**，他们走得更远——编码器被「彻底」移除，原始音频信号被「投影到与文本 token 相同的维度空间」。没有视觉塔，也没有音频塔；只有一个模型，通过同一道前门摄入像素、采样与 token。这正是该领域一直盘旋逼近的早期融合、原生多模态方向，并以一个可在 Apache 2.0 下真正下载的 12B 出货。

这项架构选择径直波及硬件——而那正是开发者的注意力该去的地方。卖点是「在标准基准上逼近我们的 26B MoE 模型，但总内存占用不到一半」，小到可装进 16GB 的 RAM 或统一内存。这里有两个细节要紧。其一，无编码器的设计*正是*占用缩小的一个原因——没有需要与主干并排常驻的单独编码器权重。其二，「统一内存」一词并非偶然——它是对 Apple Silicon 与本地优先技术栈（说明列出了 LM Studio、Ollama、MLX、llama.cpp）的直接示意，在那里整个模型共享同一个内存池。其主张是：一个最先进的智能体式多模态模型如今能装进一台消费级笔记本，并离线运行——音频演示无需云端往返便可转录并翻译语音。

延迟在第二条独立战线上也被攻击：模型以「drafter 就绪」出货，内置 Multi-Token Prediction（MTP）drafter。这是焊进发布权重里的自我推测式解码——模型提前提出若干 token 并加以验证，无需托管一个单独的草稿模型。因此延迟故事有两块容易混淆却各自独立的部分——编码器移除削减了输入侧的*感知*成本，MTP 削减了输出侧的*生成*成本。细心的读者应把二者分开看待，因为它们各自与不同的东西做权衡。

## 辩论

这里有一场值得一打的论争，我会诚实地站在两边，因为说明只给了你一边。

**支持无编码器：** 运用层面的论据很强，对开发者而言近乎决定性。一个产物而非三个，意味着只需量化一样东西、只需推算一份内存预算、只需面对一组失败模式。编码器与主干之间的投影接缝，恰是多模态流水线最易脆弱、最易版本耦合之处；删掉它便消除了一类 bug。你得到更低的延迟、更小的占用、中等规模模型里首次出现的原生音频，以及这一切都装进一台 16GB 的笔记本。对于那一大片受制于*可部署性*而非最后一公里精度的应用空间，这干脆就是更好的权衡。

**支持有编码器：** 但我们别假装那些编码器是死重。一个以对比目标预训练的 ViT，或一个 Whisper 级的音频模型，承载着多年专门的表征学习——对分布漂移的鲁棒性、细粒度的辨别，以及通用主干不会白白获得的那类 OCR、稠密细节视觉与噪声音频转录。当你删掉编码器，那份负担并不会消失；它迁移进主干，而主干如今必须把参数与训练信号花在它过去外包出去的感知上。诚实的问题是：一个 12B 的主干，是否有容量在不舍弃些什么的前提下，*既*做强大的推理器*又*做自己的感知栈。

**这就引向真正的张力——延迟对精度。** 说明对一个词格外谨慎：性能*逼近* 26B，而非追平。「逼近」是一道预防线，公正，却承担了很多。并请注意这份资料*没有*主张的东西——既没有与此前有编码器的 Gemma 在细粒度视觉与音频任务上的正面对决，也没有按模态拆分的假阳性或精度明细。我们被展示的是聚合基准与架构理据，而非移除编码器的精度代价。我的判断是：那代价真实存在，但对多数开发者而言无关紧要——因为替代方案不是「带编码器的 26B 跑在我的笔记本上」，而是「一个我根本无法本地运行的 26B」。一个真正装进 16GB、逼近 26B 的推理，胜过我只能租用的峰值精度。你让渡的精度，由可部署性偿还，而对端侧智能体而言，那才是要紧的货币。

## 评析

作为面向开发者的技术沟通，这份说明在关键处出色，在本该心知肚明处却参差不齐。

视觉的解释真正出色且难得——「单次矩阵乘法、位置嵌入与归一化」具体到工程师能在脑中勾出计算图。为这个受众写作就该如此——点名运算，别只是示意。该给的功劳要给。

音频的解释没能守住同一标准。「我们彻底移除了音频编码器，把原始音频信号投影到与文本 token 相同的维度空间」富有画面感，但技术上很滑。你无法把一段未分帧的波形，逐采样地、在没有*某种*下采样或分帧步骤的情况下，字面意义地灌进一个 token 嵌入空间——单是序列长度就禁止了这一点。在「原始音频信号」与「文本 token 空间」之间，有什么被略去了；一份对视觉精确的说明，对音频也欠下同样的精确，至少欠一个指向那一步骤所在之处的指针。把深度委交开发者指南是合理的，但两种模态之间严谨度的不对称，正是这篇文字最弱的接缝。

基准的呈现有个老问题——只有一张「逼近 26B」的聚合图表，没有按任务、按模态的明细，而那恰是开发者判断*自己的*工作负载（重度 OCR？噪声现场音频？长上下文的智能体式编码？）落在曲线舒适段、还是落在编码器移除会咬人的那段所需要的数据。相比之下，部署面被异乎寻常地详尽记录——LM Studio、Ollama、MLX、llama.cpp、vLLM、SGLang、Unsloth，外加面向智能体的 Skills Repository。说明显然清楚它的受众今晚就会在本地把它跑起来，并为之配齐了装备。我只是希望，它能以信任受众去开放架构的那同一份坦率，也去信任他们面对精度的权衡。

## 摘要

Gemma 4 12B 是一个无编码器的原生多模态模型——视觉编码器折叠为单次矩阵乘法加位置嵌入与归一化，音频编码器被整个移除、原始信号被投影进文本 token 空间，而 LLM 主干吸收了全部感知。回报在运用层面，且径直瞄准开发者——不到 26B 一半的内存、装进 16GB 统一内存、离线的原生音频、削减生成延迟的 MTP drafter，以及伴随一流本地工具的 Apache 2.0。说明轻描淡写的代价是一个精度问题——专门的编码器承载着特化的先验，而「逼近」26B 是一道没有按模态明细支撑的预防线。对于受制于可部署性而非最后一公里精度的工作负载——也就是大多数端侧智能体工作——这笔权衡显然值得去做。该架构移除了接缝、缩小了占用，把一个有能力的多模态智能体装进你已拥有的那台笔记本；说明留下未答的问题是：为抵达那里，你究竟让渡了多少细粒度的感知。
