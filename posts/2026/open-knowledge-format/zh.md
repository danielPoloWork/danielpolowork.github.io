三个月前，我把 Mycelium OS——我为 AI 智能体打造的本地、保护隐私的记忆层——的知识层，构建成了一个 markdown 文件目录，每个文件顶部带着几个结构化字段。我以为自己是在即兴发挥。然后 Google Cloud 的 Data Cloud 团队发布了 **Open Knowledge Format（开放知识格式）**，我才意识到，我一直在手写一份尚未发表的标准。本文把 OKF v0.1（Sam McVeety 与 Amir Hormati，2026 年 6 月）当作它所声称的东西来读——不是产品，而是一种*格式*——并追问：对每一个与碎片化的企业知识搏斗的人来说，这个区别意味着什么。

## 深入解析

先从论点说起，因为这是多数团队搞错的地方：束缚智能体系统的约束不是模型能力，而是上下文。一个能写代码、能总结文档、能分析数据集的模型，若没有你的表结构、你那项指标的业务含义、两套系统之间的连接路径，照样会自信满满地输出垃圾。OKF 的主张是：缺失的那一块是**一种格式，而不是又一个服务**。这一句就是全部论证，而且它是对的。

就机制而言，OKF 简单得近乎令人发噱。一个*捆绑包（bundle）*就是一个 markdown 文件目录，每个文件是一个*概念*——一张表、一个数据集、一项指标、一份运行手册、一个 API。文件路径就是概念的身份。一小块 YAML frontmatter 承载可查询字段（仅 `type` 为必填；`title`、`description`、`resource`、`tags`、`timestamp` 皆可选），markdown 正文承载其余一切。概念之间用普通的 markdown 链接相连，于是目录变成一张*图*，比文件系统所暗示的父子树更丰富。可选的 `index.md` 为智能体在层级中导航时提供渐进式披露，可选的 `log.md` 提供按时间排列的历史。v0.1 规范的全部内容只占一页。

有意思的是它强制带来的那个对比——文件对数据库，这才是表层之下真正的主题。元数据目录把知识建模为藏在 API、SDK 和专有 schema 背后的行。OKF 把它建模为 git 中的文件。我们诚实地把后果一一列出：

- **可读性。** 一个 markdown 概念，在任何编辑器里都能被人读懂，*同时*无需翻译层就能被智能体解析——同一个文件，没有序列化这一步。而一行目录记录，无论人还是智能体，在读取之前都需要一次查询、一个客户端、凭证以及一次反序列化。对智能体分析而言这是决定性的：文件本身就是你投进上下文窗口的*载荷*。数据库在模型看到任何东西之前，都需要一次工具调用、对 schema 的了解，以及一次往返。
- **可移植性。** 文件能在系统间迁移而幸存——一个 tarball、一个 git 仓库、一个挂载的文件系统。而目录里的知识被锁在创造它的那个界面背后，导出本身就是一个项目。
- **谱系。** 知识与它所描述的代码并排存于版本控制之中，于是 diff、blame、PR 评审、回滚都免费到手。数据库是事后再加审计日志；git 原生就给你来源谱系。
- **生产者与消费者相互独立。** 格式即契约，两端的工具可随意替换。人工手写的捆绑包可被智能体读取；导出流水线生成的捆绑包可在可视化器里浏览；一个 LLM 合成的捆绑包可被另一个 LLM 查询。

文件*尤其对智能体*占优，还有一个更深的理由，它颠覆了数十年的成见。历史上数据库胜过平面文件的理由是维护：人类无法让一大堆文本文件保持一致。OKF 所依托的 Karpathy 的观察是：维护者已不再是人。「LLM 不会厌倦，不会忘记更新交叉引用，一次就能改动 15 个文件。」正是那种把人类维基逼死的记账工作，恰恰是智能体所擅长的。于是，曾经偏向数据库的维护性论证，在策展者变成智能体的那一刻便反转了。

用数据分析师来落到实处，因为价值正是在那里着陆。今天，一个问「如何从事件流计算周活跃用户」的分析师，要从互不兼容的界面里拼出答案——这边一个目录 API，那边一个维基，一段 docstring，还有某位资深工程师的记忆。在专有目录里，分析师被绑死在那家厂商的查询模型与 SDK 上，知识出不了这栋楼。换成 OKF 捆绑包，分析师——或者他的智能体——读 `metrics/weekly_active_users.md`，顺着链接到 `tables/orders.md`，看到连接路径，全是纯文本、在 git 中、无需搭建任何集成。知识跨过了厂商的边界，因为它从一开始就不依赖那个厂商。

这也是对孤岛问题的回答，而孤岛正是这一切之下的企业之痛。一个组织的知识原子散落在目录、维基、共享盘、代码注释以及少数几个人的脑袋里。每个做智能体的人都从零再解一遍上下文拼装；每个目录厂商都把同一套数据模型重造一遍；知识仍被困在生产它的东西里。一种共享格式溶解了孤岛，因为一个团队的捆绑包无需翻译就能被另一个团队的智能体消费。孤岛从来不是知识的属性——它是互不兼容界面的属性。

关于实现复杂度也说一句，因为这是它可能真正铺开的安静原因——几乎没有复杂度。没有压缩方案，没有运行时，不需要 SDK，只有一个必填字段，规范一页纸。复杂度被刻意推到了两端——Google 提供了一个富集智能体，它遍历某个 BigQuery 数据集，为每张表起草一个概念（随后第二遍 LLM 处理用引用、schema 和连接路径来充实每一个），还有一个单文件静态 HTML 可视化器，无需后端、不让任何数据离开页面，以及三个可立即浏览的样例捆绑包。格式本身则不含任何机械装置。这份克制，正是它的设计。

## 辩论

第一个真正的争论是**灵活性对结构刚性**，它就坐在那个唯一的必填字段上。极简派的理由很强：一种格式靠有多少方说它来赢得采用，每多一个必填字段，既是入门门槛，也是一个让标准永远拖延的分歧点。刚性派的理由同样强：「最小限度的主张」会滑向「无用地欠规定」。若一个生产者输出 `type: BigQuery Table`，另一个输出 `type: table`，第三个输出 `type: RelationalTable`，消费者仍然需要一个翻译层——而那正是 OKF 立志要废除的。没有语义互操作的语法互操作，只是半座桥。规范诚实地声明它定义的是「互操作的表面，而非内容模型」，但互操作真正栖身之处恰恰是*内容模型*。v0.1 靠把那场硬仗往后推，买下了眼前的采用。

第二个争论是哲学层面的，也是我最在意的——**开放的格式对封闭的平台**。专有目录卖的是集成——它替你拼装上下文，藏在一块玻璃后面，带着支持与 SLA。OKF 押的是相反的赌注：价值来自有多少方说这门格式，而不是来自谁拥有它，而且它「永远不会要求专有账号或 SDK」。管理层的本能通常偏向平台——一个厂商、一份合同、一个可以追责的对象。但平台把你的知识绑在那个厂商的寿命与价目表上；开放的格式什么也不绑，知识比任何单一工具活得都久。把它说成一项管理收益而非工程偏好：开放的格式把知识共享从一次采购决策变成一条工程惯例。每个团队都能贡献和消费，无需购买席位。于是知识共享按惯例而非按许可证扩展——而惯例是免费的。

不点名第三种张力就太天真了：作为超大规模厂商的 Google，正在发布一份厂商中立、无需 SDK 的标准，同时又「更新 Google Cloud 的 Knowledge Catalog 使其能摄取 OKF」。愤世嫉俗地读，这是「把互补品商品化」——让格式免费，让最擅长供应它的上层平台胜出。宽厚地读，这格式确实会比作者的商业利益活得更久。两种读法同时为真，而其化解之道正是 OKF 所宣讲的那条原理：格式可与动机相分离。把生产者/消费者的独立性应用到 OKF 自己的作者身上，意味着我们既能采用这份标准，又能清醒地看清它为何被白白送出。

最后一点是私人的，但我把它当作证据而非轶事奉上。我为 Mycelium OS 独立地抵达了几乎一模一样的形态——markdown 概念、frontmatter、交叉链接、用于渐进式披露的索引——彼时我并不知道 LLM 维基这一模式有名字，更不知道它有规范。当一种模式被从未协调过的人们各自重新发明，那是「这模式承重而非赶时髦」所能给出的最强信号。它也把真正的风险挪了位置：形态没有疑问——大家本就在向它收敛——有疑问的是惯例。采用*格式*几乎免费；就*词汇*达成一致才是真正的工作。

## 评析

它的优点是真实的，值得直说，因为稀有。「是格式，不是平台」是正确的框架，而几乎没人这样出货；业界的默认值是再来一个服务。把整件事扎根于 git 与 markdown，务实到了优雅的程度——它搭乘已经存在的工具（diff、PR 评审、全文检索），而不去重造它们。而「一页规范、一个必填字段」的纪律本身就是美德——能出货的欠规定，胜过永远出不了货的过度规定。

我会施压之处：

1. **语义互操作是尚未解决的核心。** 没有共享的类型词汇——或至少一套命名空间与别名惯例——消费者就会一直翻译，那句头条承诺便会漏水。我想要一个*推荐*（而非必填）的类型注册表，以及一个高于「拥有 `type` 字段」的合规级别。
2. **规模与检索。** 「目录即捆绑包」在数千个概念时很美；到几十万时，它就撞上了数据库之所以存在的理由——索引、查询规划、部分加载。OKF 把渐进式披露押在 `index.md` 上，却没有给出超越「智能体在层级中导航」的大规模检索方案。实践中你会在其之上构建一层索引或嵌入——这没问题，但格式应当把这一点大声说出来。我的 Mycelium 正撞上了这堵墙，并用 markdown 之上的确定性分块与检索来回应，不用向量数据库；这是格式正确地保持开放、却讨论不足的消费者侧关切。
3. **写入协调。** `log.md` 是对历史的优雅致意，但两个智能体编辑同一个捆绑包，是规范未触及的合并冲突问题。git 提供机制；OKF 不提供协调写入的惯例。
4. **来源与信任。** frontmatter 承载 `timestamp` 与 `resource`，却没有作者身份、验证状态或陈旧度——而当一个智能体即将*依据*知识去*行动*时，这恰是消费者所需的字段。「自动生成、从未经人工审核」应当是可表达的。
5. **富集智能体是双刃剑。** 它证明了 OKF 能被机器规模化地生产，但自动起草的概念，其可信度只等同于充实它的那第二遍 LLM 处理。没有来源字段，一束貌似可信却未经验证的概念，就是一份披着资产戏服的负债。

这些都不致命。它们是 v0.2 到 v1.0 的议程，而规范明确地为向后兼容的成长做了版本化。正确的姿态是现在就采用，并把惯例往前推——而不是去等一份按其自身设计、永远不会以完整形态到来的成品标准。

## 摘要

- 智能体系统的约束是上下文，不是能力。OKF 的主张——缺失的那一块是格式，不是又一个服务——是对的。
- OKF v0.1 是一个带 YAML frontmatter（仅 `type` 必填）的 markdown 概念目录，文件路径即身份，markdown 链接即图。只是 markdown，只是文件，只是 YAML。
- 对阵专有目录，文件在可读性（人与智能体共用一个文件，无需翻译）、可移植性（git、tarball、挂载）与谱系（免费的版本控制）上占优。在规模与检索上数据库仍占优，所以预期要加一层消费者侧索引。
- 核心权衡是灵活性对刚性：一个必填字段把采用最大化，却把更难、更有价值的问题——语义互操作——往后推。
- 哲学核心是开放的格式对封闭的平台：价值积累于有多少方说这门格式，而非谁拥有它。对管理层而言，这把知识共享从一次许可决策变成一条工程惯例——按惯例而非按席位扩展。
- 私人注脚：在 OKF 公开存在之前，我已经在 Mycelium OS 里运行了这套一模一样的模式三个月。独立的重新发明，是「模式承重」的最强证据。采用这份格式；并出一份力，去修补它仍留待解决的那些惯例。
