“整洁架构”是那种人人引用、却少有人诚实践行的概念。我想把著名的同心圆图删繁就简、只留下承重的主张，为其中经得起真实代码考验的两三条辩护，并直说哪些部分纯属仪式。

## 深入解析

在那些圆环之下，真正值得记住的规则只有一条：**依赖指向内部，指向稳定的策略。** 其余一切都是它的推论。

- **边界**把因不同原因而变化的东西分开。数据库因基础设施的原因而变，定价规则因业务的原因而变。在它们之间放一道接缝。
- **依赖方向**才是全部：核心定义接口，外层实现它们。你的领域永远不 import 你的 Web 框架；是 Web 框架 import 你的领域。
- **不依赖框架的核心**就是回报。当业务逻辑在编译期对 HTTP、ORM、消息总线一无所知时，它便能多年保持可测试、可移植、可读。

就这些。三个想法，其中一个——方向——蕴含了另外两个。

## 辩论

真正的张力是**纯粹派对务实派**。

- *纯粹派*在每一个边界都放一个端口和一个适配器，在每一层之间放一个映射器，每一次跨越都配一个 DTO。承诺是彻底解耦。
- *务实派*反驳说，每一层间接都是要写、要读、要调试的代码；而多数应用永远不会更换数据库，所以这种抽象是对一件永不发生之事的保险。

我的立场：对*方向*要近乎虔诚地坚持，对*仪式*要定量配给。端口值得设在你确实预期会变、或需要在测试中替身的边缘——数据库、第三方 API、时钟。而在两个总是一起变化的层之间，它就是死重。架构关乎你为哪些接缝付费，而非把接缝最大化。

## 评析

那套经典讲法把自己卖得过头了，这点值得说出来。图示暗示你需要四层同心圆、并在每个边界都做一次转换；实践中这会催生“改一个字段要动六个文件”的代码库。这套材料若能以依赖规则开篇、把圆环当作*一种可能的图示*而非强制规范，并大声申明间接是有成本、需要挣得的，就会更好。整洁架构最强的版本，对删除不承重的抽象最为冷酷。

## 摘要

整洁架构可以归结为一个经久的检验：**你能否在不触碰核心逻辑的情况下，替换数据库、框架或交付方式？** 如果能，那么无论你画了多少圆环，这套架构都足够整洁。在变化原因分叉处保留边界，让每一个依赖都指向内部，并把每一个映射器和端口都当作必须证明其价值的成本。纪律在于方向，而不在于图。
