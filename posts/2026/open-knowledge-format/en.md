Three months ago I built the knowledge layer for Mycelium OS — my local, privacy-preserving memory for AI agents — as a directory of markdown files, each carrying a few structured fields at the top. I thought I was improvising. Then Google Cloud's Data Cloud team published the **Open Knowledge Format**, and I realized I had been writing an unpublished standard by hand. This piece reads OKF v0.1 (Sam McVeety and Amir Hormati, June 2026) as exactly what it claims to be — not a product, but a *format* — and asks what that distinction means for anyone fighting fragmented enterprise knowledge.

## Deep dive

Start with the thesis, because it's the part most teams get wrong: the binding constraint on agentic systems is not model capability, it's context. A model that can write code, summarize a document, or analyze a dataset still produces confident garbage without the schema of your table, the business meaning of your metric, the join path between two systems. OKF's claim is that the missing piece is **a format, not another service**. That sentence is the whole argument, and it is correct.

Mechanically, OKF is almost insultingly simple. A *bundle* is a directory of markdown files, each file a *concept* — a table, dataset, metric, runbook, API. The file path is the concept's identity. A small block of YAML frontmatter carries the queryable fields (only `type` is required; `title`, `description`, `resource`, `tags`, `timestamp` are optional), and the markdown body carries everything else. Concepts link to each other with ordinary markdown links, which turns the directory into a *graph* richer than the parent/child tree the filesystem implies. Optional `index.md` files give agents progressive disclosure as they navigate; optional `log.md` files give chronological history. The full v0.1 spec fits on one page.

The interesting move is the comparison this forces — files versus a database, which is the real subject under the surface. A metadata catalog models knowledge as rows behind an API, an SDK, and a proprietary schema. OKF models it as files in git. Line up the consequences honestly:

- **Readability.** A markdown concept is legible to a human in any editor *and* parseable by an agent with no translation layer — the same file, no serialization step. A catalog row needs a query, a client, credentials, and a deserialization before either a human or an agent can read it. For agentic analysis this is decisive: the file *is* the payload you drop into a context window. A database needs a tool call, schema knowledge, and a round trip before the model sees anything.
- **Portability.** Files survive moving between systems — a tarball, a git repo, a mounted filesystem. A catalog's knowledge is locked behind whichever surface created it, and exporting it is a project.
- **Lineage.** Knowledge lives in version control next to the code it describes, so diff, blame, PR review, and rollback come for free. Databases bolt audit logs on afterward; git gives provenance natively.
- **Producer/consumer independence.** The format is the contract and the tooling at each end is swappable. A bundle hand-authored by a human can be read by an agent; a bundle generated by an export pipeline can be browsed in a visualizer; a bundle synthesized by one LLM can be queried by another.

There is a deeper reason files win *specifically for agents*, and it inverts decades of received wisdom. The historical case for databases over flat files was maintenance: humans cannot keep a sprawl of text files consistent. Karpathy's observation, which OKF builds on, is that the maintainer is no longer human. "LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass." The exact bookkeeping that kills human wikis is what agents are good at. So the maintenance argument that used to favor the database reverses the moment the curator is an agent.

Make it concrete with the data analyst, because that's where the value lands. Today an analyst asking "how do I compute weekly active users from our event stream?" assembles the answer from mutually incompatible surfaces — a catalog API here, a wiki there, a docstring, the memory of one senior engineer. With a proprietary catalog the analyst is bound to that vendor's query model and SDK, and the knowledge cannot leave the building. With an OKF bundle the analyst — or their agent — reads `metrics/weekly_active_users.md`, follows the link to `tables/orders.md`, sees the join path, all in plain text, in git, with no integration to stand up. The knowledge has crossed the vendor boundary because it never depended on the vendor.

This is also the answer to the silo problem, which is the enterprise pain underneath all of it. An organization's knowledge atoms live scattered across catalogs, wikis, drives, code comments, and a few people's heads. Every agent builder re-solves context assembly from scratch; every catalog vendor reinvents the same data model; the knowledge stays trapped in whatever produced it. A shared format dissolves the silo because one team's bundle is consumable by another team's agent without translation. The silo was never a property of the knowledge — it was a property of incompatible surfaces.

A word on implementation complexity, because it is the quiet reason this might actually spread: there almost isn't any. No compression scheme, no runtime, no required SDK, one mandatory field, a one-page spec. The complexity is deliberately pushed to the edges — Google ships an enrichment agent that walks a BigQuery dataset and drafts a concept per table (then a second LLM pass enriches each with citations, schemas, and join paths), a single-file static HTML visualizer that needs no backend and lets no data leave the page, and three ready-to-browse sample bundles. The format itself stays empty of machinery. That restraint is the design.

## Debate

The first real argument is **flexibility versus structural rigor**, and it sits on that single required field. The minimalist case is strong: a format earns adoption by how many parties speak it, and every additional required field is both a barrier to entry and a point of disagreement that delays the standard forever. The rigorist case is equally strong: "minimally opinionated" shades into "uselessly underspecified." If one producer emits `type: BigQuery Table`, another `type: table`, and a third `type: RelationalTable`, consumers still need a translation layer — which is precisely what OKF set out to abolish. Syntactic interoperability without semantic interoperability is half a bridge. The spec is honest that it defines "the interoperability surface, not the content model," but the content model *is* where interoperability actually lives. v0.1 buys adoption now by deferring the hard fight.

The second argument is the philosophical one, and it's the one I care about most: **open format versus closed platform**. A proprietary catalog sells integration — it assembles the context for you, behind one pane of glass, with support and an SLA. OKF makes the opposite bet: value comes from how many parties speak the format, not from who owns it, and it "will never require a proprietary account or SDK." Management's reflex usually favors the platform — one vendor, one contract, one throat to choke. But the platform binds your knowledge to that vendor's lifespan and price list; the open format binds nothing, and the knowledge outlives any single tool. Stated as a management benefit rather than an engineering preference: an open format turns knowledge-sharing from a procurement decision into an engineering convention. Every team can contribute and consume without buying seats. Knowledge sharing then scales with conventions, not with licenses — and conventions are free.

It would be naive not to name the third tension: Google, a hyperscaler, is publishing a vendor-neutral, no-SDK standard, and in the same breath "updating Google Cloud's Knowledge Catalog to ingest OKF." Read cynically, that is commoditize-the-complement — make the format free so the platform that serves it best wins the layer above. Read generously, the format genuinely outlives the author's commercial interest. Both readings are true at once, and the resolution is the very principle OKF preaches: the format is separable from the motive. Producer/consumer independence, applied to OKF's own author, means we can adopt the standard while staying clear-eyed about why it was given away.

The last point is personal, and I offer it as evidence rather than anecdote. I reached almost exactly this shape independently for Mycelium OS — markdown concepts, frontmatter, cross-links, an index for progressive disclosure — without knowing the LLM-wiki pattern had a name, let alone a specification. When a pattern is reinvented by people who never coordinated, that is the strongest possible signal that it is load-bearing rather than fashionable. It also relocates the real risk: the shape is not in question — everyone is already converging on it — the conventions are. Adoption of the *format* is nearly free; agreement on the *vocabulary* is the work.

## Critique

The strengths are real and worth stating plainly, because they are rare. "Format, not platform" is the correct framing and almost nobody ships it; the industry default is one more service. Grounding the whole thing in git and markdown is pragmatic to the point of elegance — it rides tooling that already exists (diff, PR review, full-text search) instead of reinventing it. And the discipline of a one-page spec with a single required field is a virtue: under-specification that ships beats over-specification that never does.

Where I would push:

1. **Semantic interoperability is the unsolved core.** Without a shared type vocabulary — or at minimum a namespacing and aliasing convention — consumers keep translating, and the headline promise leaks. I'd want a *recommended* (not required) type registry and a conformance level beyond "has a `type` field."
2. **Scale and retrieval.** A bundle-as-directory is beautiful at thousands of concepts; at hundreds of thousands it meets the reasons databases exist — indexing, query planning, partial loading. OKF leans on `index.md` for progressive disclosure, but offers no story for retrieval at scale beyond "the agent navigates the hierarchy." In practice you will build an index or embedding layer on top — which is fine, but the format should say so out loud. My Mycelium work hit exactly this wall and answered it with deterministic chunking and retrieval over the markdown, no vector database; that is a consumer-side concern the format correctly leaves open but under-discusses.
3. **Write coordination.** `log.md` is a graceful nod to history, but two agents editing one bundle is a merge-conflict problem the spec doesn't address. Git supplies the mechanism; OKF supplies no convention for coordinating writes.
4. **Provenance and trust.** Frontmatter carries `timestamp` and `resource` but nothing for authorship, verification status, or staleness — exactly the fields a consumer needs when an agent is about to *act* on the knowledge. "Auto-generated, never human-reviewed" should be expressible.
5. **The enrichment agent cuts both ways.** It proves OKF can be machine-produced at scale, but auto-drafted concepts are only as trustworthy as the second LLM pass that enriched them. Without provenance fields, a bundle of plausible-but-unverified concepts is a liability wearing the costume of an asset.

None of this is fatal. It is the v0.2-to-v1.0 agenda, and the spec is explicitly versioned for backward-compatible growth. The correct posture is to adopt now and push the conventions forward rather than wait for a finished standard that, by its own design, will never arrive fully formed.

## Summary

- The constraint on agentic systems is context, not capability. OKF's thesis — the missing piece is a format, not another service — is right.
- OKF v0.1 is a directory of markdown concepts with YAML frontmatter (only `type` required), file path as identity, markdown links as graph. Just markdown, just files, just YAML.
- Against proprietary catalogs, files win on readability (one file for human and agent, no translation), portability (git, tarball, mount), and lineage (version control for free). Databases still win on scale and retrieval, so expect to add a consumer-side index layer.
- The central tradeoff is flexibility versus rigor: one required field maximizes adoption but defers semantic interoperability — the harder, more valuable problem.
- The philosophical core is open format versus closed platform: value accrues from how many parties speak the format, not from who owns it. For management that converts knowledge-sharing from a licensing decision into an engineering convention — it scales with conventions, not seats.
- Personal note: I ran this exact pattern in Mycelium OS for three months before OKF existed in public. Independent reinvention is the strongest evidence a pattern is load-bearing. Adopt the format; help fix the conventions it still leaves open.
