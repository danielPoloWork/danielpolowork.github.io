这份资料里最具启示的一行，是末尾那句安静的话——「我们将很快关停 Chat 模式」。Mistral 的 **Vibe** 公告，把 Le Chat 改名为一个智能体，并借此宣告：聊天机器人的时代已被折叠进智能体的时代。但真正该让一位从业者停下的细节，不是改名——而是那张功能清单读起来有多么*眼熟*。技能（Skills）、子智能体、自定义模式、会话作用域的权限、先计划后执行、共享同一套线束（harness）的 CLI 与 IDE 扩展。这一个月我一直在读智能体的发布，而这一篇告诉我——这个品类已经停止发明界面，转而开始标准化它。

## 深入解析

把功能目录当作一个单一主张来读，论题便浮现了——编码与工作的智能体如今有了一套*参考架构*，而 Vibe 是它的一个忠实实例。看看这里出货了什么。把可重复工作流变成 `/` 命令的可重用**技能**。在会话内为专门工作分流的**子智能体**与**自定义模式**。一个由智能体提交以供签字（批准）、可在运行前编辑、并能在运行中提出多选问题的**计划**。带有对文件、命令、目录之覆盖的「总是 / 从不 / 询问」**会话作用域权限**。一个隔离的**沙箱**、并行且持久的**会话**、用于上下文的 `@` 提及，以及把活动会话在终端与云之间搬移的 `/teleport`。CLI 与 VS Code 扩展背后的同一套**线束**。这些没有一个为 Vibe 所独有——它们合在一起，如今是智能体式编码工具的*事实上的*语法，正如标签页、断点与文件树成为了 IDE 的语法。那次收敛才是故事，而它是一个健康的迹象——当一个界面稳定下来，竞争便从新颖性移开，转向真正会累积的东西——模型质量、执行可靠性，以及围绕权限的信任模型。

两种模式之间的运用分割，是设计变得有趣之处，而那个追问二者如何不同的简报是对的。**Work 模式**是水平的——它跨越 Google Workspace、Outlook、SharePoint、Slack、GitHub 而抵达，用 Canvas 起草交付物，把一份上传的电子表格分析成会话内的图表，调度周期性运行，并把结果推送到 Notion 或你的收件箱。**Code 模式**是垂直的——它连接 GitHub，在一个隔离的沙箱里运行会话，并把一项变更一路带到一个可审阅的拉取请求。同一个智能体在 Web 应用、VS Code 侧边栏与终端里都可用。但要紧的差别不是水平还是垂直——而是*安全闸门的形状*，而二者以一种公告并未推到前台的方式，是非对称的。

Code 模式有一道内建于其工作流的、自然而强大的审阅闸门——智能体在沙箱里工作，你在它写的当口检查 diff，而在一个人类合并那个拉取请求之前，没有任何东西抵达生产。它在构造上是可逆的。Work 模式没有等价的闸门。它作用于你的*活动*系统——一个真实的收件箱、一个真实的 SharePoint、一个真实的 Notion——而它唯一的检查点，是你在开始前签字的那份事先计划。批准一个计划，与批准那些行动并不相同；而一个写着「发送更新」的计划，会变成一封你撤不回的邮件。所以那个更安静、更重大的观察是这样的——人们本能上最害怕的编码智能体，在设计上其实是两者中*更安全的*那个，因为软件工程早已发明了审阅闸门（PR），而 Vibe 继承了它。那个感觉起来更温和的知识工作智能体，反而携带着更大的、不可逆的波及半径。

还有两条线索合拢了这个环。**技能**是最被低估的功能，因为它是「工作流即代码」——把一个可重复的流程一次性地、作为一个受版本管理的 `/` 命令、以「一致性与顺序上的精确」被执行地编码下来，之于即兴提示，正如函数之于复制粘贴。对个人，那是便利——对组织，那是治理：同一个工作流被所有人以相同方式运行，可审计、被标准化。那才是真正的大组织效率杠杆，而非人均的提速。而它与 Mistral *另一个*同日发布的连接并非偶然——Vibe 的「企业知识搜索」跨越所有那些连接器而扎根，正是 Search Toolkit 为之存在以求解决的那个检索问题。一家公司，在同一个发布周期里，出货了那个智能体，以及它赖以进食的检索基底。那是一个首尾一贯的平台赌注——智能体不会好过它所能检索到的东西，这条论点贯穿每一个值得审视的智能体系统。

## 辩论

**像 Vibe 这样的智能体，真的会降低生产力吗？** 简报把它问得很挑衅，而诚实的答案是「有时会，且这取决于一个公告从不度量的量——审阅成本」。Vibe 把人类从*做*移向*监督*——Work 模式的每一次运行都把步骤流式呈现给你观看，Code 模式的每一个 diff 都邀请检查，每一份计划都想要签字。那是为信任而设的正确设计，但它并非免费。净生产力只有在「监督这个智能体」比「你自己做这项任务」更便宜时才为正，而那个不等式随任务而反转。对那条可自动化的长尾——追上一个收件箱、起草一份模板化报告、运行一次被调度的数据抽取——监督是微不足道的，而收益是现实而巨大的。对判断密集的工作，验证一个貌似有理却错误的交付物的成本，会逼近乃至超过产出它的成本，而且你还承接了*没能抓住*那个错误的风险。智能体并不消除工作——它把著述转换成审阅。而那究竟是不是收益，是一个逐任务的经验问题，而非一句口号。

**权限的安全性有多稳健？** 模型本身是合理的——会话作用域的总是 / 从不 / 询问、文件与命令的覆盖、沙箱隔离、管理员级的组织控制。风险不在机制，而在那个*被聚合的面*。一个「扎根于你的团队早已使用的文档、邮箱与系统」的 Work 模式智能体——同时是 Workspace、Outlook、SharePoint、Slack、GitHub——在构造上，是公司里被授予权限最广的行为者，横向地伸进每一个孤岛。那恰恰是严肃的智能体安全研究当作核心威胁来对待的、那个被过度授权的内部者——一个拥有正当而宽广的访问、其行为你无法完全预测的实体。会话作用域的「询问」是一种真实的控制——直到批准疲劳降临、用户为了止住提示而点下「总是」为止；而计划对行动的鸿沟，意味着最强的 Work 模式检查点批准的是意图，而非后果。安全设计是称职的——波及半径，是没被处理的那一部分。

**它让一个大组织更高效吗？** 总体而言是，但其机制是标准化，而非速度。一个跨工作与代码的智能体、一份许可，减少了工具蔓延——共享的技能把最佳实践工作流制度化——管理员权限把治理中央集权化。代价是它的镜像——你已经把知识工作集中到单一厂商的智能体身上，并造出了一个如今是你资产中价值最高之标靶的权限面。效率与集中风险，是同一个架构决定的两面。

## 评析

作为一次发布，沟通是清晰的、分层得当的、对自身形状诚实的——Work/Code 的分割很干净，每一步可见的设计被平实地陈述，而 Le Chat 用户的迁移故事被处理得不挥手敷衍。对开发者，它以具体、而非形容词，回答了那些要紧的问题（CLI/IDE 的对等、沙箱、PR 流、技术栈连接器）。那是一篇做得很好的公告。

它的隙缝有两处。其一，它断言了它本该度量的回报。「把一上午的管理事务变成一个提示」是一个鲜活的主张，也恰恰是那种需要一个数字的生产力断言——省下的时间、无人值守完成的任务、自主运行的错误率——而它一个也没有。对一个把长程自主当作全部卖点的智能体而言，缺席的指标是*长程上的可靠性*——一次多步运行在无人介入下正确收尾的频率有多高？那是买家所需的唯一数字，也是没被给出的那个。

其二，Work 模式的安全非对称，值得被直视，而非被留作暗示。作品仔细地描述了 Code 模式的沙箱与 diff 检查——那条可逆的路径——而 Work 模式对活动系统的不可逆行动，只得到事先的计划签字，文档却从不承认这些是不同的风险等级。一个更严谨的版本会平实地说——Code 模式由合并来把守，Work 模式只由计划来把守，而我们如此遏制那道差别。而在二者之下，那个收敛点转为批评而坐下——当每一件严肃的工具都共享这套界面时，公告真正的差异化要素是模型质量与运行可靠性，作品点了它的名（「为智能体任务优化的旗舰 Mistral 模型」），却不加实证。界面如今是入场的底注——而它背后的引擎能把活儿干完的证据，才是缺席的那样东西。

## 摘要

Vibe 把 Le Chat 改名为一个统一的工作与代码智能体，并在关停 Chat 模式之际，把聊天机器人当作这个领域已经走过的一个阶段来对待。最有用的观察是结构性的——Vibe 的功能集——作为 `/` 命令的技能、子智能体、自定义模式、会话作用域的「总是 / 从不 / 询问」权限、可编辑的计划、被沙箱化的并行会话、CLI 与 IDE 的对等、`/teleport`——如今是智能体式编码工具的*事实上的*参考架构，而向一个共享界面的收敛意味着竞争已移向模型质量、可靠性与信任设计。在运用上，Work 模式（水平、跨连接器、作用于活动系统）与 Code 模式（垂直、被沙箱化、由拉取请求把守）之别，与其说在范围，不如说在安全的形状——编码智能体是*更安全的*那个，因为工程早已发明了审阅闸门，而知识工作智能体，在仅有事先计划签字的背后，携带着更大的不可逆波及半径。在生产力上，诚实的裁决是：Vibe 把著述转换成监督——对可自动化的长尾是真实的收益，对判断密集的工作则是打平或更糟，在那里验证一个貌似有理却错误的结果，花的与亲手去做一样多。它的技能是「工作流即代码」，对大组织的真正价值是标准化与治理，而非生的速度；而它宽广的连接器触达，使它同时既是效率的胜利，又是每一个严肃的智能体安全分析所警告的、被过度授权的内部者。这次发布清晰、对开发者可信，却伏下了最要紧的两个数字——长程运行的可靠性，以及那个如今已被商品化的界面背后的模型证据。
