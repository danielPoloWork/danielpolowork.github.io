多智能体系统在图上很优雅，在生产中却很混乱。搭建过几套之后，总有那么几个模式反复证明自己的价值，也总有几个以同样的方式失败。

## 先扇出，再收敛

最简单且有用的形态是 **扇出／收敛**：把任务拆成相互独立的单元，并发执行，再合并结果。

- 每个单元对其他单元一无所知 — 利于覆盖，不利于一致性。
- 真正的设计在合并步骤：去重、排序、冲突消解。

> 只有当下一阶段确实需要 *全部* 上游结果时，屏障（同步）才是合理的。否则，用流水线。

## 对抗式校验

“看似合理却错误”的答案是代价最高的失败模式。最便宜的保险，是派出以**反驳**为职责的独立怀疑者，只有在多数未能反驳时才保留结论。

```js
const votes = await Promise.all(
  reviewers.map((r) => r.refute(claim))
);
const survives = votes.filter((v) => !v.refuted).length >= 2;
```

## 真正会出问题的地方

1. **静默截断** — 悄悄限制工作量，会被读成“已全部覆盖”。
2. **收敛死循环** — 要对 *见过的全部* 去重，而不仅是已确认的结果。
3. **上下文膨胀** — 智能体返回散文而非结构化数据。

能存活下来的模式，是刻意的“无聊”：小单元、明确契约，以及一个你能讲清楚的合并步骤。
