三か月前、私は Mycelium OS ——AI エージェントのためのローカルでプライバシーを保つ記憶層—— の知識レイヤーを、先頭にいくつかの構造化フィールドを持つマークダウンファイルのディレクトリとして構築した。自分では即興でやっているつもりだった。ところが Google Cloud の Data Cloud チームが **Open Knowledge Format** を公開し、私は未発表の標準を手で書いていたのだと気づいた。本稿は OKF v0.1（Sam McVeety、Amir Hormati、2026年6月）を、それが名乗るとおりのもの——製品ではなく*フォーマット*——として読み解き、断片化した企業ナレッジと格闘するすべての人にとってその区別が何を意味するのかを問う。

## 深掘り

まず主張から始めよう。多くのチームが取り違える部分だからだ。エージェント型システムを縛る制約はモデルの能力ではなく、コンテキストである。コードを書け、文書を要約し、データセットを分析できるモデルでも、テーブルのスキーマ、メトリックのビジネス上の意味、二つのシステム間の結合経路がなければ、自信たっぷりにゴミを出力する。OKF の主張は、欠けているのは**サービスではなくフォーマットだ**というものだ。この一文が議論のすべてであり、そして正しい。

仕組みとしての OKF は、ほとんど拍子抜けするほど単純だ。*バンドル*はマークダウンファイルのディレクトリであり、各ファイルが一つの*コンセプト*——テーブル、データセット、メトリック、ランブック、API——を表す。ファイルパスがコンセプトの同一性そのものになる。小さな YAML フロントマターがクエリ可能なフィールドを担い（必須は `type` のみ。`title`、`description`、`resource`、`tags`、`timestamp` は任意）、マークダウン本文がそれ以外のすべてを担う。コンセプト同士はふつうのマークダウンリンクで結ばれ、それによってディレクトリは、ファイルシステムが示す親子ツリーよりも豊かな*グラフ*になる。任意の `index.md` はエージェントが階層をたどる際の段階的開示を与え、任意の `log.md` は時系列の履歴を与える。v0.1 仕様の全体は一ページに収まる。

興味深いのは、これが強いる比較——ファイル対データベース——であり、これこそ表層の下にある本当の主題だ。メタデータカタログは知識を、API・SDK・独自スキーマの背後にある行としてモデル化する。OKF はそれを git の中のファイルとしてモデル化する。帰結を正直に並べてみよう。

- **可読性。** マークダウンのコンセプトは、どんなエディタでも人間が読め、*かつ*翻訳レイヤーなしにエージェントが解析できる——同じファイル、シリアライズの手順なし。カタログの行は、人間にせよエージェントにせよ読む前に、クエリ・クライアント・認証情報・デシリアライズを要する。エージェント分析にとってこれは決定的だ。ファイルはそのままコンテキストウィンドウに投入する*ペイロード*になる。データベースはモデルが何かを見る前に、ツール呼び出し・スキーマ知識・往復通信を要する。
- **可搬性。** ファイルはシステム間の移動を生き延びる——tarball、git リポジトリ、マウントされたファイルシステム。カタログの知識はそれを生んだ面に閉じ込められ、エクスポートは一つのプロジェクトになる。
- **来歴。** 知識はそれが記述するコードの隣でバージョン管理下に置かれるので、差分・blame・PR レビュー・ロールバックがただで手に入る。データベースは監査ログを後付けする。git は来歴をネイティブに与える。
- **生産者と消費者の独立。** フォーマットが契約であり、両端のツールは差し替え可能だ。人間が手書きしたバンドルをエージェントが読め、エクスポートパイプラインが生成したバンドルをビジュアライザで閲覧でき、ある LLM が合成したバンドルを別の LLM がクエリできる。

ファイルが*とりわけエージェントにとって*勝る、より深い理由があり、それは数十年の通念を反転させる。フラットファイルよりデータベースを支持してきた歴史的根拠は保守性だった——人間は散らばったテキストファイル群を一貫させられない。OKF が土台にする Karpathy の観察は、保守する主体がもはや人間ではない、というものだ。「LLM は退屈せず、相互参照の更新を忘れず、一度のパスで15個のファイルに触れられる。」人間のウィキを殺すまさにその記帳作業こそ、エージェントが得意とすることだ。つまり、かつてデータベースを有利にした保守性の議論は、キュレーターがエージェントになった瞬間に逆転する。

データアナリストで具体化しよう。価値が着地するのはそこだからだ。今日、「イベントストリームから週間アクティブユーザーをどう計算するか」と問うアナリストは、互いに非互換な面から答えを組み立てる——こちらにカタログ API、あちらにウィキ、docstring、そして一人の上級エンジニアの記憶。独自カタログではアナリストはそのベンダーのクエリモデルと SDK に縛られ、知識は社外に出られない。OKF バンドルなら、アナリスト——あるいはそのエージェント——は `metrics/weekly_active_users.md` を読み、`tables/orders.md` へのリンクをたどり、結合経路を見る。すべて平文で、git の中で、立ち上げるべき統合なしに。知識がベンダーの境界を越えたのは、そもそもベンダーに依存していなかったからだ。

これはサイロ問題への答えでもあり、それがすべての根底にある企業の痛みだ。組織の知識の原子は、カタログ・ウィキ・共有ドライブ・コードコメント・数人の頭の中に散らばっている。エージェントを作る者は皆コンテキスト組み立てを一から解き直し、カタログベンダーは皆同じデータモデルを作り直し、知識はそれを生んだものの中に閉じ込められたままになる。共有フォーマットはサイロを溶かす。あるチームのバンドルは、別のチームのエージェントが翻訳なしに消費できるからだ。サイロは知識の性質ではなかった——非互換な面の性質だったのだ。

実装の複雑さについて一言。これが実際に広まりうる静かな理由だからだ——複雑さがほとんど無い。圧縮方式なし、ランタイムなし、必須 SDK なし、必須フィールドは一つ、仕様は一ページ。複雑さは意図的に両端へ押し出されている——Google は、BigQuery データセットを巡回してテーブルごとにコンセプトを起草する（そして二度目の LLM パスが引用・スキーマ・結合経路で各々を補強する）エンリッチメントエージェント、バックエンド不要でデータをページ外に出さない単一ファイルの静的 HTML ビジュアライザ、そしてすぐ閲覧できる三つのサンプルバンドルを提供する。フォーマット自体は仕掛けを持たない。その抑制こそが設計なのだ。

## ディベート

最初の本当の論点は**柔軟性対構造的厳格さ**であり、それはあの唯一の必須フィールドの上に乗っている。ミニマリストの言い分は強い。フォーマットは何者がそれを話すかの数で採用を勝ち取るのであり、必須フィールドが増えるたびに参入障壁となり、標準を永遠に遅らせる対立点になる。厳格主義者の言い分も同じく強い。「最小限にしか意見を持たない」は「役に立たないほど未規定」へと滑り落ちる。ある生産者が `type: BigQuery Table` を、別の生産者が `type: table` を、三人目が `type: RelationalTable` を出力すれば、消費者はやはり翻訳レイヤーを必要とする——それこそ OKF が廃そうとしたものだ。意味的相互運用性を欠いた構文的相互運用性は、半分だけの橋である。仕様は「コンテンツモデルではなく相互運用の表面を定義する」と正直に述べるが、相互運用性が実際に宿るのは*コンテンツモデル*の側だ。v0.1 は難しい戦いを先送りにすることで、今の採用を買っている。

二つ目の論点は哲学的なものであり、私が最も気にかけるものだ——**開かれたフォーマット対閉じたプラットフォーム**。独自カタログは統合を売る——あなたの代わりにコンテキストを組み立て、一枚のガラスの背後で、サポートと SLA 付きで。OKF は逆に賭ける。価値は誰が所有するかではなく何者がフォーマットを話すかから来るのであり、それは「独自アカウントや SDK を決して要求しない」。経営の反射は通常プラットフォームを好む——一ベンダー、一契約、責任を問う相手が一つ。だがプラットフォームはあなたの知識をそのベンダーの寿命と価格表に縛りつける。開かれたフォーマットは何も縛らず、知識はいかなる単一ツールより長く生きる。エンジニアの好みではなく経営上の便益として言い換えれば——開かれたフォーマットはナレッジ共有を、調達の決定からエンジニアリングの慣習へと変える。どのチームも席を買わずに貢献し消費できる。ナレッジ共有はライセンスではなく慣習に応じてスケールする——そして慣習は無料だ。

三つ目の緊張を名指ししないのは素朴すぎる。ハイパースケーラーである Google が、ベンダー中立で SDK 不要の標準を公開し、同じ息で「Google Cloud の Knowledge Catalog を OKF を取り込めるよう更新」している。皮肉に読めば、これは補完財のコモディティ化だ——フォーマットを無料にし、それを最も上手く供する上位レイヤーのプラットフォームが勝つ。寛大に読めば、フォーマットは著者の商業的利害よりも本当に長く生きる。両方の読みが同時に真であり、その解決は OKF が説くまさにその原理だ——フォーマットは動機から切り離せる。生産者と消費者の独立を OKF 自身の著者に適用すれば、なぜそれが手放されたのかを冷静に見据えつつ標準を採用できる。

最後の点は個人的なものだが、逸話ではなく証拠として差し出す。私は Mycelium OS のためにほぼ正確にこの形——マークダウンのコンセプト、フロントマター、相互リンク、段階的開示のためのインデックス——に独立して到達した。LLM ウィキのパターンに名前があることも、ましてや仕様があることも知らずに。誰も連携していない人々によってパターンが再発明されるとき、それはそのパターンが流行ではなく荷重を支えているという、可能なかぎり最も強い信号だ。それはまた本当のリスクの所在を移す。形は問われていない——皆すでにそこへ収束しつつある——問われているのは慣習だ。*フォーマット*の採用はほぼ無料であり、*語彙*の合意こそが仕事である。

## 批評

強みは本物であり、はっきり述べる価値がある。稀だからだ。「プラットフォームではなくフォーマット」は正しい枠組みであり、ほとんど誰もそれを出荷しない。業界の既定値はもう一つのサービスだ。全体を git とマークダウンに根づかせたのは、優雅さに達するほど実際的だ——既に存在するツール（差分、PR レビュー、全文検索）に乗り、それを作り直さない。そして必須フィールド一つの一ページ仕様という規律は美徳だ——出荷される過少規定は、決して出荷されない過剰規定に勝る。

私が押すであろう点：

1. **意味的相互運用性が未解決の核心だ。** 共有された型語彙——少なくとも名前空間とエイリアスの慣習——がなければ、消費者は翻訳し続け、看板の約束は漏れる。私は*推奨*（必須ではない）型レジストリと、「`type` フィールドを持つ」以上の適合レベルが欲しい。
2. **規模と検索。** ディレクトリとしてのバンドルは数千コンセプトでは美しい。数十万になれば、データベースが存在する理由——索引付け、クエリ計画、部分ロード——に出会う。OKF は段階的開示を `index.md` に頼るが、「エージェントが階層をたどる」以上の規模での検索の物語を持たない。実際にはその上に索引や埋め込みのレイヤーを作ることになる——それでよいが、フォーマットはそれを声に出して言うべきだ。私の Mycelium はまさにこの壁にぶつかり、ベクトルデータベースなしのマークダウン上の決定論的チャンキングと検索で答えた。これはフォーマットが正しく開けたままにしている消費者側の関心事だが、論じ方が足りない。
3. **書き込みの調整。** `log.md` は履歴への優雅な目配せだが、一つのバンドルを二つのエージェントが編集することは、仕様が扱わないマージ衝突の問題だ。git は仕組みを供するが、OKF は書き込みを調整する慣習を供さない。
4. **来歴と信頼。** フロントマターは `timestamp` と `resource` を担うが、著者性・検証状態・陳腐化については何も持たない——エージェントが知識に*基づいて行動*しようとするとき、消費者が必要とするまさにそのフィールドだ。「自動生成、人間未レビュー」が表現できるべきだ。
5. **エンリッチメントエージェントは両刃だ。** それは OKF が機械によって規模を持って生産できることを証明するが、自動起草されたコンセプトは、それを補強した二度目の LLM パスと同じだけしか信頼できない。来歴フィールドがなければ、もっともらしいが未検証のコンセプトの束は、資産の衣装をまとった負債だ。

どれも致命的ではない。これは v0.2 から v1.0 への議題であり、仕様は後方互換の成長のために明示的にバージョン管理されている。正しい姿勢は、いま採用し、慣習を前へ押し進めることだ——その設計上、決して完全な形では到来しない完成標準を待つのではなく。

## 要約

- エージェント型システムの制約は能力ではなくコンテキストだ。OKF の主張——欠けているのはサービスではなくフォーマット——は正しい。
- OKF v0.1 は、YAML フロントマター（必須は `type` のみ）を持つマークダウンコンセプトのディレクトリであり、ファイルパスが同一性、マークダウンリンクがグラフ。ただのマークダウン、ただのファイル、ただの YAML。
- 独自カタログに対し、ファイルは可読性（人間とエージェントに一つのファイル、翻訳なし）、可搬性（git、tarball、マウント）、来歴（ただのバージョン管理）で勝つ。規模と検索ではデータベースがなお勝つので、消費者側の索引レイヤーを足すことになる。
- 中心的なトレードオフは柔軟性対厳格さだ。必須フィールド一つは採用を最大化するが、より難しくより価値ある問題——意味的相互運用性——を先送りする。
- 哲学的な核心は開かれたフォーマット対閉じたプラットフォーム。価値は誰が所有するかではなく何者がフォーマットを話すかから積み上がる。経営にとってそれはナレッジ共有を、ライセンスの決定からエンジニアリングの慣習へと変える——席ではなく慣習に応じてスケールする。
- 個人的な注記：私は OKF が公に存在する前の三か月間、まさにこのパターンを Mycelium OS で運用していた。独立した再発明は、パターンが荷重を支えていることの最も強い証拠だ。フォーマットを採用し、それがなお開けたままにしている慣習を直す手伝いをしよう。
